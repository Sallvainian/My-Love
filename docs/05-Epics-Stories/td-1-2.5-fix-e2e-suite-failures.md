## Dev Notes

### Root Cause Analysis

The 36 failing tests fall into these categories:

1. **Mock Route Conflicts**: Tests register routes with `page.route()` but don't clean them up. Subsequent tests in the same spec inherit stale routes.

2. **Fixture Shadowing**: `love-notes.setup.ts` defines its own `navigateToLoveNotes` (line 134) which shadows the merged fixtures version, causing different navigation behavior.

3. **State Leakage**: Tests modify localStorage/Zustand state that persists to subsequent tests.

4. **Anti-Pattern Propagation**: `.catch(() => false)` patterns in fixture files cause non-deterministic test behavior.

### Test Infrastructure Summary

**Key Files:**
- `tests/support/fixtures/index.ts` - Merged fixture entry point
- `tests/support/fixtures/baseFixture.ts` - Legacy cleanApp, appWithMessages fixtures
- `tests/support/fixtures/app-fixtures.ts` - New `$`, `loveNotes`, `navigateToLoveNotes` fixtures
- `tests/support/selectors/` - Strict selector patterns

**Fixture Hierarchy (from tests/support/fixtures/index.ts):**
```
test = mergeTests(
  networkErrorMonitorFixture,  // Auto-fail on 4xx/5xx      ← lowest precedence
  logFixture,                  // Structured logging
  baseFixture,                 // cleanApp, appWithMessages  ← .catch anti-patterns HERE
  monitoredFixture,            // Console/network monitoring
  dataFixture,                 // loveNoteFactory
  appFixtures                  // $, loveNotes, navigation   ← .catch anti-pattern HERE (line 184)
);                             //                            ↑ highest precedence
```

**⚠️ Fixture Precedence (CRITICAL):**
- `mergeTests()` uses LAST fixture for conflicts (last wins)
- `love-notes.setup.ts` does its own `test.extend()` AFTER importing from merged fixtures
- This means `love-notes.setup.ts:navigateToLoveNotes` **OVERRIDES** `appFixtures.navigateToLoveNotes`
- Tests importing from love-notes.setup.ts use a DIFFERENT navigation implementation!

**Network-First Pattern (CRITICAL):**
```typescript
// CORRECT: Route BEFORE navigate
await page.route(pattern, handler);
const responsePromise = page.waitForResponse(matcher);
await navigateToLoveNotes();
await responsePromise;

// WRONG: Navigate then route
await navigateToLoveNotes();
await page.route(pattern, handler); // Too late!
```

### Primary Failure Pattern (Validated 2025-12-10)

Error stack trace points to `love-notes.setup.ts:146`:
```
Error: page.waitForResponse: Test ended.
at love-notes/love-notes.setup.ts:146
```

**Root Cause (CORRECTED):**
Line 146 is inside the `navigateToLoveNotes` fixture DEFINITION (not a test call). The failure occurs because:

1. **Fixture Shadowing**: `love-notes.setup.ts` defines its own `navigateToLoveNotes` (lines 128-165) which SHADOWS the merged fixtures version
2. **Tests use wrong fixture**: Tests importing from love-notes.setup.ts get a different `navigateToLoveNotes` than tests using merged fixtures
3. **Response promise setup**: The shadowed fixture's `waitForResponse()` (line 146) races with mock routes that may not be registered

**Why This Causes Suite Failures:**
- When run individually, tests may pass if they have proper `beforeEach` route setup
- When run as suite, previous tests' route handlers leak into subsequent tests
- The shadowed fixture expects a specific response that stale routes don't provide

**Resolution Path:**
1. Task 2.5: Remove or rename the shadowed fixture in love-notes.setup.ts
2. Task 7: Ensure all tests use merged fixtures' `navigateToLoveNotes`
