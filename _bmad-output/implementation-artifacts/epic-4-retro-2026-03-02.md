## Key Insights

1. **Error handling is an architecture, not a feature** — Console logging is not error handling. A production-ready error pipeline needs structured categorization, Sentry integration, and bounded retry with backoff. This must be built as infrastructure, not sprinkled per-function.

2. **Channel lifecycle needs defensive coding** — Supabase Realtime channels can fail on send, fail on cleanup, and silently disconnect. Every channel operation needs `.catch()` or try/catch, and the presence channel needs a CLOSED state handler.

3. **State reset granularity matters** — `...initialScriptureState` is a shotgun reset that destroys derived state. State resets should be scoped to the specific state being invalidated, not the entire slice.

4. **Version guards must be first** — Any operation that checks version/phase must do so before any state mutation. The `end_session` ordering bug (C3) is a textbook race condition: state is reset before the version check can prevent it.

5. **External code review adds orthogonal value** — 9 internal review rounds caught structural and pattern issues. Two external LLM reviewers caught systemic architectural gaps (error pipeline, retry bounds, auth patterns) that internal reviews missed. Both are needed.

6. **4 review rounds signals foundational gaps** — Story 4.3's record review count wasn't just complexity — it was building reconnection logic on top of incomplete error handling from 4.1/4.2. Fix the foundation before extending it.

## Systemic Themes

The PR review findings clustered into 4 systemic themes:

### Theme 1: Error Handling Pipeline

No structured error reporting exists. `handleScriptureError` logs to console only. Channel operations are fire-and-forget. Sentry is not wired in.

### Theme 2: Reconnection Failure Modes

Reconnection retries have no cap and no backoff. Channel cleanup has no error handling. Presence channel has no CLOSED state handler.

### Theme 3: State Ordering Under Stress

Version checks happen after state mutations. State resets are overly broad. String matching on error codes is fragile.

### Theme 4: Auth Identity Verification

`auth.getUser()` errors are unchecked. `currentUserId` can be null and cascade silently through broadcast and RPC calls.
